> Fuckoff-Gov — слабо централизованный мессенджер с E2E-шифрованием и групповыми чатами, написанный при помощи Go + Fyne.

> https://habr.com/ru/articles/1014982/

## Установка

```bash
$ go install github.com/number571/fuckoff-gov@latest
```

## Запуск приложения

```bash 
$ fuckoff-gov --database {{database-file}}
# [--database] = путь к файлу встраиваемой bbolt БД (по умолчанию client.db)
```

## Поддерживаемые платформы

- Windows (x86_64, arm64)
- Linux (x86_64, arm64)
- Android (x86_64, arm64)

## Модель угроз

Мессенджер Fuckoff-Gov не является анонимным. Внешний наблюдатель может располагать информацией о взамиодействии клиентов с сервером и на основе этого делать дальнейшие предположения об отправлении и получении сообщений. Компрометация сервера позволит наблюдателю получить более точную и обширную информацию о пользователях, такую как: инициализация / подгрузка клиента, инициализация / подгрузка канала, отправление / получение сообщения, список клиентов, список участников канала, список зашифрованных сообщений в канале. Для анонимизации канала рекомендуется использовать анонимную сеть, а через мессенджер передавать симметричные / публичные ключи, если того требует процедура инициализации анонимизирующей системы.

Мессенджер Fuckoff-Gov не защищает от наблюдателя, у которого имеется доступ к устройству. Предполагается, что среда, в которой будет запущен процесс мессенджера, является надёжной и не допускает к себе сторонних лиц. Если чувствуется угроза, то следует удалить приложение и воспользоваться каким-либо механизмом очистки памяти. В крайнем случае следует физически ликвидировать устройство, а конкретно - модули памяти.

Мессенджер Fuckoff-Gov защищает передаваемые сообщения от чтения сторонними лицами посредством симметричного шифрования (AES-256-GCM). Ключ симметричного алгоритма генерируется на стороне создателя канала и передаётся в шифрованном виде на публичном ключе каждого участника канала (ML-KEM-768). Каждое передаваемое сообщение и инициализация канала подписывается приватным ключом создателя (ML-DSA-65). Для создания хеш-значений и кодов аутентификации используется хеш-функция SHA-384. Параметры были выбраны аналогично анонимной сети Hidden Lake и указаны в работе [hidden_lake_anonymous_network.pdf](https://github.com/number571/hidden-lake/blob/master/docs/hidden_lake_anonymous_network.pdf). Все они приведены к 128-битной безопасности, при условии существования достаточно мощных квантовых компьютеров, что делает криптографические примитивы безопасными. Мессенджер аналогично наследует свойство квантовой устойчивости.

## Протокол

### Инициализация клиента

```
1. (sk, pk) <- G()          // генерация приватного / публичного ключей
2. p <- P(pk)               // подтверждение публичного ключа доказательством работы
3. (p, pk) -> SERVER        // регистрация публичного ключа на сервере
```

### Подгрузка клиентов

```
1. H(req_pk) -> SERVER      // запрос на получение публичного ключа по хешу
2. (p, pk) <- SERVER        // получение ключа с доказательством работы
3. ok ?= Vp(pk, p)          // проверка доказательства работы
4. H(req_pk) ?= H(pk)       // проверка полученного публичного ключа
5. pk -> STORAGE            // сохранение ключа
```

### Инициализация канала

```
1. k <- G()                                                 // ключ канала
2. enc_n <- E(k,name)                                       // зашифрованное название канала
3. PL = (H(my_pk), H(pk1), H(pk2), ..., H(pkN))             // хеши публичных ключей участников канала
4. EL = (E(my_pk, k), E(pk1, k), E(pk2, k), ..., E(pkN, k)) // зашифрованный ключ канала публичными ключами участников
5. id <- HMAC(k, (enc_n || PL || EL))                       // идентификатор канала
6. s <- S(my_sk, id)                                        // подпись канала
7. p <- P(id)                                               // подтверждение идентификатора канала доказательством работы
8. (id, enc_n, PL, EL, p, s) -> SERVER                      // отправление информации о канале серверу
```

### Подгрузка каналов

```
1. (id, enc_n, PL, EL, p, s) <- SERVER  // получение канала
2. ok ?= Vs(PL[0], id, s)               // проверка подписи
3. ok ?= Vp(id, p)                      // проверка доказательства работы
4. index <- H(my_pk) ?in PL             // проверка моего ключа в списке
5. k <- D(my_sk, EL[index])             // получение ключа канала
6. id ?= HMAC(k, (n || PL || EL))       // проверка идентификатора канала
7. n <- D(k, enc_n)                     // получение названия канала
8. (id, k, n, PL) -> STORAGE            // сохранение канала
```

### Отправление сообщения в канал

```
1. k <- STORAGE[id]                     // получение ключа канала 
2. c <- E(k, m)                         // создание шифртекста
3. h <- HMAC(id, c)                     // хеш шифртекста
4. s <- S(my_pk, h)                     // подпись хеша
5. p <- P(h)                            // подтверждение хеша доказательством работы
6. (id, H(my_pk), c, s, p) -> SERVER    // отправление сообщения на сервер 
```

### Получение сообщения из канала

```
1. (id, H(pk), c, s, p) <- SERVER   // получение сообщения от сервера
2. ok ?= Vs(pk, c, s)               // проверка подписи участника
3. ok ?= Vp(c, p)                   // проверка доказательства работы
4. (k, PL) <- STORAGE[id]           // получение ключа и списка участников канала 
5. H(pk) ?in PL                     // проверка принадлежности участника к каналу
6. m <- D(k, c)                     // расшифрование сообщения
7. m -> STORAGE                     // сохранение сообщения
```
